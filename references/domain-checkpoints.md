# 学科专项检查点

优先选择最贴近论文主贡献的一类检查点。对交叉型论文，可选 1 个主类型和 1 个辅类型联合审阅，但不要把所有类型的要求机械叠加。

## 算法 / 模型类

重点检查：

- 是否给出明确的问题定义、输入输出、优化目标与适用前提。
- 是否与足够强且足够新的基线做公平对比。
- 是否提供消融实验、敏感性分析、复杂度分析或失败案例。
- 是否交代数据集划分、预处理、超参数、随机性控制与评估指标。
- 是否把工程技巧、数据清洗优势或额外先验错误包装成“方法创新”。

常见风险：

- 基线偏弱或调参不公平。
- 只报告最优结果，不报告波动范围或重复实验。
- 核心模块缺少必要的消融与解释。
- 结论完全依赖单一数据集或单一场景。

## 系统 / 平台 / 软件工程类

重点检查：

- 是否存在完整的系统目标、架构、模块边界、接口与数据流闭环。
- 是否有真实工作负载、性能指标、可靠性、扩展性或安全性测试。
- 是否说明部署环境、软硬件栈、关键配置与瓶颈分析。
- 是否展示系统相对已有方案的实用改进，而非仅完成课程设计式实现。
- 是否把“能运行”与“可支撑论文结论”混为一谈。

常见风险：

- 系统功能堆砌，但缺少主线问题和关键技术难点。
- 只有功能截图，没有系统级指标或对照实验。
- 模块设计复杂，但没有解释为何必须如此设计。
- 测试样例过小，无法支撑工程价值结论。

## 硬件 / 电路 / 器件类

重点检查：

- 是否说明设计约束、指标目标、工艺条件、器件参数和测试平台。
- 是否比较仿真结果与实测结果，并解释偏差来源。
- 是否给出功耗、面积、速度、精度、稳定性或良率等关键指标。
- 是否在相同或可比工艺/条件下与已有方案进行公平比较。
- 是否分析误差来源、温漂、噪声、寄生效应或制造偏差。

常见风险：

- 只有仿真，没有实测或样机验证。
- 比较对象工艺条件不同，却直接给出优越性结论。
- 测试环境、仪器精度和误差来源交代不足。
- 关键器件或模块没有实际工作波形、频谱或版图级证据。

## 通信 / 信号 / 控制类

重点检查：

- 是否清楚写出信道模型、系统模型、控制对象或扰动假设。
- 是否给出核心公式推导、稳定性/收敛性/鲁棒性分析。
- 是否进行多工况、多噪声或多参数区间的验证。
- 是否包含仿真与实物/半实物/硬件在环中的至少一种强验证。
- 是否公平比较传统方法、经典方法与近年代表性方法。

常见风险：

- 假设条件过强，脱离实际工况。
- 只展示单组理想参数下的结果。
- 理论分析与实验设置脱节。
- 鲁棒性、实时性或复杂度没有交代。

## 数据应用 / 交叉工程类

重点检查：

- 是否清楚说明数据来源、标注质量、隐私合规性与场景边界。
- 是否证明方法确实解决了领域问题，而不是只提升了离线指标。
- 是否交代领域约束、部署条件、误报漏报成本或资源限制。
- 是否分析泛化能力、数据偏差、类别不平衡与分布漂移。
- 是否说明与真实业务流程、实验平台或应用闭环的关系。

常见风险：

- 只做离线指标比较，缺少应用闭环评价。
- 忽略数据采样偏差或标签噪声。
- 用通用模型套领域数据，却没有领域适配分析。
- 标题宣称工程应用，正文却只有算法实验。

## 通用红旗项

看到下列情形时，应提高警惕并在评分中真实反映：

- 题目、摘要、结论与正文主线不一致。
- 参考文献陈旧且缺少近三到五年的关键工作。
- 图表无法独立支撑结论，或坐标、单位、实验条件缺失。
- 章节间重复堆砌，工作量看似很大但核心增量有限。
- 只强调“首次”“显著提升”，却没有给出严格证据。
- 局限性、失败案例和适用边界完全缺失。
